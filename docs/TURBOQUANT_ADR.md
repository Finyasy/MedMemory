# TurboQuant ADR

## Status

Proposed

## Decision

MedMemory will evaluate TurboQuant as a **KV-cache compression strategy** for MedGemma with these initial architecture decisions:

- primary local runtime target: `MLX` on Apple Silicon
- fallback/comparison runtime: `Transformers on MPS`
- Phase 1 scope: **text-generation KV cache only**
- default first profile: `3.5-bit`
- experimental profile: `2.5-bit`
- first outlier policy: **fixed, versioned partition**
- first prototype style: **correctness-first**
- startup assets: generated by a bootstrap script and loaded at startup

## Context

MedMemory already uses MedGemma locally and is constrained by long-context generation cost, especially on Apple Silicon.

TurboQuant is relevant because it targets:

- KV-cache compression
- online inference
- low distortion under constrained bit-width
- improved long-context efficiency

The paper supports the general direction, but it does not describe:

- MLX-specific cache interception
- MedGemma-specific runtime integration
- FastAPI lifecycle integration
- the exact outlier-selection rule for this repo

## Why This Decision

### 1. MLX-first is the right local target

This repo is being evaluated on a MacBook. Local acceptance should therefore be decided on the same hardware and the same runtime path that will actually be used during development.

`MLX` is the preferred target because:

- it is the natural Apple Silicon local runtime
- it aligns with the repo's existing local runtime direction
- it is the only baseline that matters for local decision-making on this machine

`Transformers on MPS` remains useful as:

- a fallback
- a comparison path
- a rescue path if MLX KV-cache interception is not clean enough

### 2. Phase 1 must stay narrow

Phase 1 is limited to `text-generation KV cache`.

It does not include:

- multimodal generation
- retrieval/index compression
- document extraction
- speech
- training-time weight quantization

This keeps the first implementation tied to the strongest and most validated use case from the paper.

### 3. 3.5-bit is the right first target

`3.5-bit` is the default first target because it is the safest balance between:

- compression
- quality retention
- implementation ambition

`2.5-bit` remains experimental because it is more likely to require:

- stronger outlier tuning
- `TurboQuantprod`
- more careful attention-fidelity validation

### 4. Fixed outlier partition is the right first policy

The paper clearly supports split-channel bit allocation, but it does not fully specify the exact outlier detector for MedGemma in this repo.

So the first implementation should use:

- a fixed partition
- versioned profile metadata
- benchmark-driven revision later

This is simpler and more reproducible than starting with an adaptive heavy-hitter policy.

### 5. Correctness-first is the right prototype philosophy

The first local prototype should optimize for:

- mathematical correctness
- observability
- bounded complexity

It should not optimize for:

- maximum runtime speed
- fused kernels
- aggressive low-bit profiles

This matters because the primary unknown is runtime integration, not paper math.

## Key Architectural Risk

The highest-risk unknown is **KV-cache interception** in the chosen runtime.

The practical question is:

- can the local MedGemma runtime intercept or substitute KV-cache writes and reads cleanly enough to store compressed payloads and recover attention-time values?

If the answer is no for `MLX`, then the first working prototype should pivot to `Transformers on MPS` without changing the benchmark plan.

## Consequences

### Positive

- keeps the implementation aligned with actual MacBook usage
- keeps the first milestone narrow and measurable
- avoids premature expansion into unrelated surfaces
- makes benchmark results locally meaningful

### Negative

- defers multimodal and retrieval compression work
- may require a runtime pivot if MLX interception is not viable
- keeps `2.5-bit` out of default scope initially

## Alternatives Considered

### 1. CUDA-first prototype

Rejected as the primary path because local acceptance happens on a MacBook, not on CUDA hardware.

### 2. Adaptive outlier detection first

Rejected for Phase 1 because it adds complexity before a fixed split policy has been benchmarked.

### 3. TurboQuantprod first

Rejected as the initial default because:

- it is more complex
- `TurboQuantmse` is sufficient for a first `3.5-bit` benchmark
- `TurboQuantprod` should be introduced when low-bit attention drift actually appears

## Immediate Next Step

Execute `Sprint 0`:

- prove or disprove clean `MLX` KV-cache interception
- freeze the cache-shape contract
- decide whether MLX remains the first working prototype target

## Review Trigger

This ADR should be revisited when one of these becomes true:

- MLX cache interception is proven infeasible
- `3.5-bit` fails local quality gates
- `2.5-bit` becomes a product requirement
- adaptive outlier handling becomes necessary
